# 回归与标定说明

这个文档专门说明当前项目的“题库导入 -> 回归验证 -> LLM 标定 -> 区间回写”流程。

如果你接下来要继续优化分数稳定性，建议按本文档执行，而不是直接改 Prompt 后只测单题。

## 1. 先理解这几类文件

当前回归体系主要依赖下面几类数据：

1. 题目 JSON
   位置：`ai_gongwu_backend/assets/questions/`
2. 回归样本文本
   位置：`ai_gongwu_backend/assets/regression_samples/`
3. 回归报告
   位置：`reports/regression/`

对 `generated_hunan` 来说，每道题通常配套：

1. 一个题目 JSON
2. 一个样本目录
3. 三个样本文本：
   - `reference_high.txt`
   - `reference_mid.txt`
   - `reference_low.txt`

## 2. 什么时候需要重新导入题库

出现下面任一情况，就应该先重新执行导入脚本：

1. 新增了题库源文档
2. 修改了 `import_hunan_question_bank.py`
3. 修改了中低档样本生成模板
4. 修改了题型识别、分档生成或样本生成规则

执行命令：

```bash
cd /home/quyu/ai_interview/ai_gongwu_backend
./venv/bin/python scripts/import_hunan_question_bank.py
```

执行后会刷新：

1. `assets/questions/generated_hunan/`
2. `assets/regression_samples/generated_hunan/`
3. `assets/questions/generated_hunan/import_summary.txt`

## 3. 为什么先跑确定性回归

确定性回归不是为了替代 LLM，而是为了先确认基础排序没有明显崩掉。

建议先跑：

```bash
cd /home/quyu/ai_interview/ai_gongwu_backend
./venv/bin/python scripts/run_regression.py
```

这一步主要检查：

1. 高档样本是否普遍高于中档样本
2. 中档样本是否普遍高于低档样本
3. 规则后处理是否把答案“拉平”了
4. 新增题型模板是否明显失真

如果这一步已经大面积失败，先不要急着跑全量 LLM 回归。

## 4. 什么时候跑真实 LLM 回归

当下面条件基本满足时，再跑真实 LLM 回归：

1. 已配置有效 `LLM_API_KEY`
2. 确定性回归排序大致合理
3. 你想验证真实模型波动，而不是只看规则链路

推荐命令：

```bash
cd /home/quyu/ai_interview/ai_gongwu_backend
./venv/bin/python scripts/run_llm_regression.py --repeat 3
```

如果只想先看子集：

```bash
cd /home/quyu/ai_interview/ai_gongwu_backend
./venv/bin/python scripts/run_llm_regression.py --question-id CS-20201212-01 --question-id HN-20200816-02 --repeat 3
```

## 5. 为什么建议固定使用 --repeat 3

真实大模型分数有波动，单次分数不稳定时，直接拿一次结果做标定很容易把区间写坏。

当前脚本的做法是：

1. 每个样本跑多次
2. 收集 `attempt_scores`
3. 取中位数作为 `actual_score`

建议默认使用：

```bash
--repeat 3
```

原因：

1. 成本比 `repeat 5` 低
2. 稳定性明显好于 `repeat 1`
3. 足够用于当前题库的日常标定

## 6. 什么时候可以使用 --writeback

只有在下面条件同时满足时，才建议回写：

1. 你已经先跑过一个小子集
2. 结果波动可接受
3. 排序没有出现明显反转
4. 你愿意把当前 LLM 行为固化为新的参考区间

命令示例：

```bash
cd /home/quyu/ai_interview/ai_gongwu_backend
./venv/bin/python scripts/run_llm_regression.py --repeat 3 --writeback
```

更稳妥的方式是先对子集回写：

```bash
cd /home/quyu/ai_interview/ai_gongwu_backend
./venv/bin/python scripts/run_llm_regression.py --question-id CS-20201212-01 --question-id CSJT-20201226-01 --repeat 3 --writeback
```

## 7. 报表里的关键字段怎么看

### 7.1 expected_range

当前样本应该落入的目标区间。

### 7.2 expectation_source

只在 LLM 回归里出现，表示当前区间来自哪一套标准：

1. `llmExpected`
   优先采用已经标定过的 LLM 区间
2. `deterministicExpected`
   如果题目还没有 LLM 区间，就退回使用初始确定性区间

### 7.3 attempt_scores

只有 `repeat > 1` 时才有，表示多次调用的原始分数列表。

它主要用来判断：

1. 模型是否抖动过大
2. 某题是否需要重新设计 Prompt 或样本区间

### 7.4 fallback_used

表示是否发生了“LLM 失败后回退到确定性评分”。

在正式做真实 LLM 标定时，通常不建议允许回退，否则会把伪成功混入结果。

### 7.5 validation_issue_count

表示后处理阶段记录了多少条校验提示。

这个值偏高通常意味着：

1. 模型输出结构不稳定
2. 证据引用质量不好
3. 某题 Prompt 还需要继续收紧

## 8. 推荐工作流

建议后续所有题库优化都按这个顺序：

1. 改导入脚本、Prompt 或规则
2. 重新导入题库
3. 跑 `run_regression.py`
4. 只挑一小批题跑 `run_llm_regression.py --repeat 3`
5. 子集稳定后再扩大范围
6. 最后才执行 `--writeback`
